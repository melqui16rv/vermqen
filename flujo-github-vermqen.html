// cómo se organizan los cambios

Idea central: En Git nunca trabajas directo en el código de producción. Primero haces una copia (rama), ahí experimentas, y cuando está listo pides permiso para integrarlo. Así el código de producción siempre funciona.

// quién hace qué

Para equipos de 3–7 personas: no todas las personas deben tener el mismo nivel de acceso. GitHub permite controlar quién puede aprobar cambios y quién puede hacer deploy.

¿Cómo configuro esto en GitHub?

Settings → Branches → Add branch protection rule

Para main: activar "Require pull request reviews" → mínimo 2. Activar "Require status checks to pass" → seleccionar el CI. Activar "Restrict who can push".

Para staging: igual pero con 1 aprobación.

Para develop: "Require pull request reviews" → 1. El CI obligatorio.
